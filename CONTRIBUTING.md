# 기여하기

English: [`CONTRIBUTING.en.md`](CONTRIBUTING.en.md)

`litomi`에 기여해 주셔서 고마워요. 이 문서는 PR을 빠르게 머지하기 위해 필요한 기준을 정리했어요.

## 행동 강령

이 프로젝트는 [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md)를 따라요.

## 빠른 시작 (개발 환경)

로컬 개발 환경 구성은 README를 먼저 따라 주세요.

- 한국어: [`README.md`](README.md)
- English: [`README.en.md`](README.en.md)

기본적으로 로컬에서 아래가 모두 떠야 해요:

- Postgres + Redis (docker compose)
- Backend
- Web

## 작업 흐름

- **이슈 먼저**: 큰 변경(리팩터/설계 변경/새 기능)은 이슈로 방향을 먼저 맞춰 주세요.
- **작게 쪼개기**: PR은 리뷰 가능한 크기로 나눠 주세요(가능하면 하나의 목적).
- **추가로 깨진 것 없게**: 관련 기능이 깨지지 않도록 최소한의 테스트를 포함해 주세요.

## 품질 기준 (필수)

- **타입 안정성**: `any`는 사용하지 말아 주세요.
- **테스트**:
  - 유닛 테스트: `bun test`
  - E2E 테스트: `bun run test:e2e`
- **정적 검사**:
  - 린트: `bun run lint`
  - 타입체크: `bun run type`

## 코드 스타일 가이드 (요약)

레포의 실제 기준은 코드와 설정이 우선이에요. 여기엔 자주 어긋나는 항목만 요약해요.

- **바렐 파일 금지**: `index.ts`로 재-export하지 말고 원본 파일을 직접 import 해요.
- **시간 상수**: 매직 넘버 대신 `ms()` / `sec()` 유틸을 사용해요.
- **SQL**:
  - 허용: `sql\`\`` (표현식/CASE 등)
  - 금지: `sql.raw(...)` 또는 문자열 결합으로 SQL 만들기
- **한국어 UX 문구**: 사용자에게 노출되는 텍스트는 **친근한 ~요 스타일**을 유지해요.

## 커밋/PR 가이드

- **커밋 메시지**: 변경 목적이 드러나게 써 주세요.
- **PR 설명**:
  - 무엇을 바꿨는지
  - 왜 바꿨는지
  - 테스트 방법(커맨드/시나리오)

## 도움이 필요하면

질문이 있으면 이슈로 올려 주세요(가능하면 재현 절차/스크린샷 포함).
